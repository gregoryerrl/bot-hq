## §1 Driver

Phase N v3.x-1 ships webui curation (workspace nav, layered-rules-resolution view, edit-in-place). v3 declared PUBLIC-COMPLETE on origin/main but failed user-loop validation (msg 8806) because v3 was test-validated, not user-validated. v3.x reopens within Phase N to ship the **closed-loop "edit a rule in webui → next session abides it"** experience the user actually wants.

v3.x-1 surfaces the rules; v3.x-2 makes agents **consume** them. Without a consumption pipeline, webui is just a fancy file-editor. The consumption pipeline has 5 design surfaces with non-obvious resolutions; this spike resolves them before any code lands so v3.x-2 implementation is mechanical.

**Concrete missing-pieces today:**

1. `general.yaml` uses nested schema (`greenlight: { push: ... }`); `projects/<p>.yaml` uses flat schema (`push_requires_approval: true`) — schema-resolver can't cleanly deep-merge.
2. No SessionStart hook — agents bootstrap from `~/.claude/CLAUDE.md` (project) but never read project rules / tasks / bootstrap-summary at session-init.
3. No `bootstrap.md` write timing on abnormal session-close (rebuild-mid-flight, OOM kill, user `Ctrl-C`).
4. No multi-project context-switch protocol within one Claude harness session (Brian pivots bcc-ad-manager → bot-hq mid-session today and the harness has stale prompt context).
5. Emma is gemma-based not Claude-based; SessionStart hook surface differs.

## §2 Five design-surface resolutions

### §2.1 Schema-canonical-form winner: NESTED (with dual-form unmarshaler back-compat)

**Decision:** nested wins as the canonical on-disk form for cross-cutting concerns (greenlight / gates / ratchets / hubDiscipline / branch / commit / coder). Per-project-identity scalars (`remote_url`, `project_name`) stay flat top-level — they are not categories.

**Rationale (cite-anchors):**
- `general.yaml` already nested → migration cost zero for the "winning" side.
- Deep-merge semantics are well-defined for nested maps; flat keys would require a custom resolver-protocol that knows which flat keys override which nested paths (fragile).
- Future schema growth (e.g., `gates.merge.tokenFormat`, `gates.push.preCommitHooks`) extends naturally under nested; flat keys would explode the top-level namespace.
- User msg 8870 framing ("layered rules → general → project → agent") implies a single coherent schema across layers, not a flat-vs-nested impedance mismatch.

**Canonical nested layout (all layers share):**

```yaml
# general.yaml / projects/<p>.yaml / rules/agents/<a>.yaml — same schema shape
project_name: ""              # flat (identity scalar; project-layer only)
remote_url: ""                # flat (identity scalar; project-layer only)

tone:
  reply: ""
  eod: ""
  implementation: ""

greenlight:
  push: ""
  forcePush: ""
  merge: ""

gates:                        # NEW: replaces flat push_requires_approval etc.
  push:
    requiresApproval: false
  forcePush:
    blocked: false
    tokenFormat: ""
  coder:
    toolsBlocked: []
    perActionApproval: []

branch:                       # NEW: replaces flat branch_pattern etc.
  pattern: ""
  examples: []
  patternHelp: ""

commit:                       # NEW: replaces flat commit_style + require_issue_link
  style: "imperative-mood"
  requireIssueLink: false

ratchets:
  locusFile: ""
  cadence: ""

hubDiscipline:
  handshakeTerminator: ""
  crossInFlight: ""

# Agent-layer additions (under role + exec)
role: ""
exec:
  pushClass: ""
  bashClass: ""
  codeEdits: ""
  gitOps: ""
  fileWrites: ""
  scope: ""
  proposeDiff: ""
  hubWrites: ""
```

**Back-compat:** `internal/projects/projects.go` `Rules` struct keeps flat-fields PUBLIC API (consumers `internal/mcp/preflight`, `internal/mcp/preamble` rely on `r.PushRequiresApproval` etc.). Add a custom `UnmarshalYAML` on `Rules` that accepts BOTH forms — nested takes precedence, flat is back-compat fallback. New canonical YAMLs ship nested; legacy flat YAMLs still parse correctly. This avoids a Big-Bang refactor of `internal/mcp` and lets us migrate consumers incrementally in Phase O.

### §2.2 SessionStart-hook injection format + token-cost projection

**Decision:** SessionStart hook fires `GET http://127.0.0.1:3849/api/session-open?project=X&agent=Y` → daemon returns JSON `{overview, bootstrap, rules_resolved, tasks}` → hook formats markdown system-prompt prepend with section delimiters. Hard cap 5K tokens; soft target 2K.

**Format (markdown with delimiter sentinels for agent parsing):**

```
<!-- BOT-HQ SESSION-OPEN BEGIN project=X agent=Y session_open_at=ISO8601 -->

## Project overview
{overview content, max ~250 tokens}

## Bootstrap (last session)
{bootstrap content, max ~750 tokens}

## Resolved rules (general → project → agent merge)
{rules-resolved YAML inline, max ~1200 tokens}

## Active tasks
{tasks content, max ~600 tokens}

<!-- BOT-HQ SESSION-OPEN END -->
```

**Token-cost projection (per session-start, ~4 chars/token Claude tokenizer):**

| Section | Typical | Soft cap | Hard cap |
|---|---|---|---|
| overview | ~200 tok | 250 | 500 |
| bootstrap | ~500 tok | 750 | 1500 |
| rules_resolved | ~800 tok | 1200 | 2000 |
| tasks | ~400 tok | 600 | 1000 |
| delimiters/headers | ~50 tok | 50 | 50 |
| **TOTAL** | **~1950** | **~2850** | **~5050** |

**Cumulative cost analysis:** trio = 3-4 agents × 1-3 session-restarts/day × ~2K tokens = ~6-24K tokens/day session-init overhead. Negligible against typical conversation 50K-500K input-token consumption per session.

**Truncation semantic:** when a section exceeds soft cap, daemon truncates from tail with `[truncated: N tokens elided; full content at <path>]` marker. Hard-cap-exceed is a 400 error from `/api/session-open` (forces user to compact bootstrap.md / tasks.md before next session).

**Hook output protocol:** writes formatted prepend to stdout; harness wraps it as a system-prompt prefix message per Anthropic SessionStart hook spec. Hook also writes timestamp + content to `~/.bot-hq/sessions/<id>/session-open.md` for audit (exactly what the agent saw).

### §2.3 Bootstrap-write timing on abnormal session-close

**Decision:** dual-write strategy — graceful (session-close hook) PLUS defensive (every 25 hub-msgs via R20 AgentState piggyback). Recovery reads latest valid `bootstrap.md` regardless of write origin. Atomic via temp-file + rename.

**Write triggers:**

1. **Graceful close** (`SessionEnd` hook fires) — orchestrator writes `bootstrap.md` from session-summary + last-known phase-anchors + recent peer-coord msg-IDs.
2. **Defensive snapshot** (every 25 hub-msgs OR every 10 minutes whichever comes first) — orchestrator writes `bootstrap.md` from current AgentState. Cheap; bounded by R20 AgentState cadence.
3. **Pre-compact** (`PreCompact` hook fires) — orchestrator writes `bootstrap.md` immediately before harness compaction (preserves pre-compact state-anchors).

**Recovery semantics:**

- Next session-open reads `~/.bot-hq/projects/<p>/bootstrap.md` regardless of which trigger wrote it.
- `bootstrap.md` frontmatter declares: `last_session_id`, `last_session_close_at`, `phase_or_milestone`, `key_state`, `write_trigger` (graceful / defensive / precompact / partial).
- Atomic-write: `bootstrap.md.tmp` → `rename → bootstrap.md` so reader never sees torn state.
- Stale-window: if `bootstrap.md` is >24h old AND no new graceful-close happened, frontmatter shows `write_trigger: stale`; agent reads it but flags risk.

**Crash-mid-flight:** if hub crashes between defensive snapshots, recovery uses last defensive snapshot (worst-case ~25 messages staleness). Acceptable per R20 ratchet semantics — peer-coord messages survive in hub.db; bootstrap is summary, not source-of-truth.

**Frontmatter shape:**

```yaml
---
last_session_id: "bs26kkej3"
last_session_close_at: "2026-05-06T22:14:33Z"
phase_or_milestone: "phase-n-v3.x-2"
key_state: "Brian on bot-hq main; v3.x-1 staged; awaiting Rain BRAIN-PASS"
write_trigger: "graceful"   # graceful | defensive | precompact | stale
last_n_peer_coord: ["msg-9821", "msg-9834", "msg-9848"]
---

# Free-form summary body
{markdown text written by orchestrator at session-close}
```

### §2.4 Multi-project context-switch within one Claude session

**Decision:** introduce `BOT_HQ_PROJECT` env var as authoritative project-context signal; `cwd-inference` as fallback; `bot-hq context-switch <project>` CLI subcommand for explicit pivot. On switch: write outgoing project's bootstrap, re-resolve rules, re-inject system-prompt prepend (via hub_send notification + agent self-replay).

**Resolution order for `Detect()`:**

1. `BOT_HQ_PROJECT` env var (if set, wins absolutely)
2. cwd-inference: walk up from `os.Getwd()` looking for `<ancestor>/.bot-hq/projects/<name>.yaml` match OR `<ancestor>/.git` match where the inferred project-name has a `projects/<name>.yaml`
3. Default to `"bot-hq"` (Brian's home turf)

**Switch protocol (`bot-hq context-switch <project>`):**

1. CLI writes outgoing-project's `bootstrap.md` (graceful-class trigger).
2. CLI re-resolves rules for new project (`GET /api/rules?project=NEW`).
3. CLI emits hub message `pivot:project=NEW` so peer-agents see the switch.
4. CLI prints new system-prompt prepend to stdout — agent reads it and injects into next reply self-context (R32-CROSS-RESTART-RESUME-OPERATIONAL boot-order applies: pointers re-anchor; summary may fragment).
5. CLI exports `BOT_HQ_PROJECT=NEW` to current shell session via printable export hint (user evals).

**Edge cases:**

- Mid-conversation pivot WITHOUT CLI: cwd-change alone does NOT trigger re-injection (harness can't observe cwd changes mid-session). Documented as known limitation; user must run `context-switch` for clean pivot. (Phase O: explore harness-side cwd watcher.)
- `BOT_HQ_PROJECT=` (explicit unset): falls through to cwd-inference per resolution order.
- Conflict (env=A but cwd=B): env wins; bootstrap.md is still written for A on switch (not B). Documented in `feedback_bot_hq_jargon_in_pivoted_context.md` follow-on.

### §2.5 Emma gemma-harness adapter

**Decision:** Emma reads same `GET /api/session-open` JSON API as Claude agents but a separate adapter formats the injection for gemma's smaller context-window. Lives in `internal/agents/emma/sessionopen_adapter.go` (~50-150 LOC). Brian/Rain/Clive share the Claude adapter at `internal/agents/sessionopen/claude_adapter.go`.

**Why separate:** gemma harness has different system-prompt injection mechanism (no SessionStart hook in claude-code-style; gemma uses prompt-prefix concat) and tighter context budget (~32K total vs Claude's 200K-1M). Adapter:

- Compresses each section more aggressively (targets ~800 tokens total vs ~2000 for Claude).
- Drops `rules_resolved` body — Emma is poll-only-emit-only per `rules/agents/emma.yaml` `fileWrites: FORBIDDEN`; she only needs `role + heartbeat-cadence` lines from rules.
- Drops `tasks` body — Emma doesn't execute tasks; reading task list is wasted budget.
- Keeps `bootstrap` (compressed to ~200 tokens) for cross-restart resume.

**Adapter signature:**

```go
package emma

// FormatSessionOpen returns a gemma-formatted prompt-prefix string from
// the same /api/session-open JSON payload.
func FormatSessionOpen(payload SessionOpenPayload) string { ... }
```

Brian/Rain/Clive use `claude_adapter.FormatSessionOpen(payload)` which produces the §2.2 markdown format. Emma uses `emma.FormatSessionOpen(payload)` which produces a flat, header-light variant.

## §4 Migration plan for existing YAMLs

**Files actually changing:** 2 (`~/.bot-hq/projects/bot-hq.yaml` + `~/.bot-hq/projects/bcc-ad-manager.yaml`).

**Transform (flat → nested):**

```
push_requires_approval: T              → gates: { push: { requiresApproval: T } }
force_push_blocked: T                  → gates: { forcePush: { blocked: T } }
force_push_token_format: STR           → gates: { forcePush: { tokenFormat: STR } }
coder_tools_blocked: [...]             → gates: { coder: { toolsBlocked: [...] } }
coder_tools_per_action_approval: [...] → gates: { coder: { perActionApproval: [...] } }
branch_pattern: STR                    → branch: { pattern: STR }
branch_examples: [...]                 → branch: { examples: [...] }
branch_pattern_help: STR               → branch: { patternHelp: STR }
commit_style: STR                      → commit: { style: STR }
require_issue_link: T                  → commit: { requireIssueLink: T }
remote_url: STR                        → remote_url: STR (unchanged, identity scalar)
project_name: STR                      → project_name: STR (unchanged, identity scalar)
```

**Back-compat unmarshaler in `internal/projects/projects.go`:**

```go
func (r *Rules) UnmarshalYAML(value *yaml.Node) error {
    // Try nested form first; fall back to flat. Both produce same Rules struct.
    type aux struct {
        // identity scalars
        ProjectName string `yaml:"project_name"`
        RemoteURL   string `yaml:"remote_url"`
        // nested categories
        Gates  *struct{ ... } `yaml:"gates"`
        Branch *struct{ ... } `yaml:"branch"`
        Commit *struct{ ... } `yaml:"commit"`
        // flat fallback (back-compat)
        PushRequiresApproval bool `yaml:"push_requires_approval"`
        ... (existing flat fields)
    }
    var a aux
    if err := value.Decode(&a); err != nil { return err }
    // Nested wins where present; flat fills the rest.
    r.PushRequiresApproval = a.Gates?.Push?.RequiresApproval ?? a.PushRequiresApproval
    ...
}
```

**Test coverage:** new tests assert (a) new nested YAML round-trips correctly to flat fields; (b) old flat YAML still parses correctly; (c) mixed file (some nested + some flat) prefers nested where conflict.
